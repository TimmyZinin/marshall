Marshall AI Listener — Спецификация дизайна дашборда

version 1.0  |  Sprint 0 (Спринт 5)  |  module: S0-F04 Dashboard Design  |  2026-03-06

Данный документ определяет полный дизайн веб-дашборда системы Marshall AI Listener: пользовательский опыт, вайрфреймы всех экранов, формальные требования к интерфейсу и компонентную модель.

1. Назначение и область применения

1.1 Цель документа

  • Пользовательские сценарии (JTBD) для каждого из 5 экранов дашборда
  • Информационная архитектура и иерархия элементов интерфейса
  • ASCII-вайрфреймы всех экранов
  • Формальные требования (REQ-DD) и критерии приёмки (AC-DD)
  • Компонентная модель, цветовая система и типографика

1.2 Аудитория документа

АудиторияРоль
Frontend-разработчик Sprint 0Основная аудитория, реализует интерфейс

1.3 Границы области применения

В области применения

  • Веб-дашборд (HTML + vanilla JS + Chart.js)
  • 5 экранов: Login, Active Trips, Alerts Feed, Trip Details, Statistics
  • 3 роли: admin, manager, viewer
  • Polling каждые 3 секунды

За пределами области применения

  • Мобильное приложение
  • Интерфейс диспетчера (работает в Telegram)
  • Административная панель управления LLM-агентом

3. Часть 1: UX-анализ

3.1 JTBD для каждого экрана

Экран Login

Основное JTBD: Когда я открываю дашборд утром или после перерыва, я хочу быстро войти под своим логином, чтобы получить доступ только к данным, соответствующим моей роли.

Доп. JTBD: При ошибке входа — понять причину. При работе на чужом ПК — сессия не оставалась активной после закрытия вкладки.

Active Trips

Основное JTBD: Когда я начинаю рабочий день, я хочу за 10 секунд увидеть все рейсы в движении и какие из них требуют внимания, чтобы расставить приоритеты на ближайший час.

Доп. JTBD: Фильтрация по заказчику. Переход к деталям по HIGH-алерту. Автообновление без F5.

Alerts Feed

Основное JTBD: Когда поступает новый HIGH-алерт, я хочу сразу увидеть что произошло, в каком рейсе и что нужно сделать, чтобы принять решение в течение 1 минуты.

Доп. JTBD: Отметить алерт как reviewed / resolved. Фильтрация по рейсу, заказчику, дате.

Trip Details

Основное JTBD: Когда у рейса есть активный алерт, я хочу увидеть полную историю сообщений из чата диспетчеров по этому рейсу, чтобы понять контекст ситуации и принять правильное решение.

Доп. JTBD: Хронологический список алертов. Копирование сообщений для отчёта.

Statistics

Основное JTBD: Когда Николя Бобров подводит итоги дня или недели, он хочет за 30 секунд увидеть: сколько рейсов было, сколько алертов, какой процент решён вовремя.

Доп. JTBD: Сравнить текущую неделю с прошлой. Провалиться в ленту алертов за конкретный день.

3.3 Информационная архитектура

3.3.1 Глобальная навигация

Логотип Marshall AI | Active Trips | Alerts Feed | Statistics | [Пользователь: email] [Выйти]

Навигация присутствует на всех экранах кроме Login. Расположена сверху (top navigation bar). Активный пункт меню подсвечивается визуально.

Дашборд
├── Login (нет nav)

├── Alerts Feed
│   └── Trip Details (дочерний экран, тот же компонент)

3.4 Пять UX-принципов

1

Критичное — всегда на виду

HIGH-алерты никогда не должны быть скрыты. На любом экране элементы с severity=HIGH отображаются с красным визуальным маркером (#ef4444). Таблицы рейсов сортируются так, чтобы рейсы с HIGH-алертами стояли первыми.

Обоснование: Метрика M7 требует, чтобы Николя понимал алерт и знал что делать в 90% случаев.

2

Актуальность без действия пользователя

Данные обновляются автоматически (polling 3 сек). Пользователь не должен нажимать F5. Индикатор последнего обновления показывает время последнего успешного запроса. При потере соединения отображается предупреждение.

Обоснование: Метрика M2 требует latency ≤5 сек.

3

Контекст всегда рядом с алертом

Каждый алерт содержит достаточно информации для первичного понимания: тип события, описание (текст от LLM), trip_id, заказчик, время. Переход к полному контексту (Trip Details) требует одного клика. Не более 2 кликов от алерта до истории чата.

Обоснование: Метрика M7 — Николя видит алерт и понимает что делать.

4

Статус-ориентированная модель данных

Каждый алерт имеет явный статус (new → reviewed → resolved), отражённый визуально. Пользователь понимает: «кто-то уже взял это в работу?» без переговоров в Telegram. Смена статуса — одна кнопка без многоуровневых диалогов.

Обоснование: В команде несколько менеджеров — без явного статуса возможны дублирование или пропуск реакции.

5

Минимальный когнитивный груз

Дашборд используется в стрессовых ситуациях. Цвет = severity (красный = HIGH, оранжевый = MEDIUM, зелёный = LOW). Иконки дублируют цвет (доступность). Тексты алертов — конкретные и actionable. Фильтры с понятными подписями, сбрасываемые одной кнопкой.

Обоснование: Cognitive overload в критической ситуации увеличивает время реакции.

12. Граничные случаи

12.1 Таблица рейсов

Случай 1: Рейс без алертов
- Строка: нормальный белый фон, нет badge-а, нет маркера ●
- Ячейка алертов: пустая или прочерк

Случай 2: Рейс с 0 active + 5 resolved алертами
- Алерты resolved не считаются «активными»
- Строка без красного фона
- Ячейка алертов: может показывать (5 решено) серым цветом

Случай 3: Рейс одновременно с HIGH и LOW
- Строка: HIGH-стиль (красный фон), badge HIGH первый
- Сортировка: данный рейс входит в HIGH-группу

Случай 4: trip_id не распознан системой (null)
- Отображается «ID не извлечён» в ячейке trip_id
- Строка кликабельна, Trip Details показывает что известно

12.2 Алерты

Случай 1: Алерт без описания от LLM (пустой текст)
- Карточка показывает тип алерта (delay, equipment_failure и т.д.)
- Вместо описания: «Подробности не извлечены системой»

Случай 2: Одновременный PATCH от двух менеджеров
- Второй PATCH получает ответ с актуальным статусом
- Polling обновит карточку через ≤3 сек
- Оптимистичное обновление откатывается если статус изменился

Случай 3: Более 100 алертов за день
- Пагинация или «Загрузить ещё» (инкрементальный скролл)
- Первые 20 алертов загружаются при открытии

12.3 Statistics

Случай 1: Нет данных за выбранный период
- KPI-карточки показывают 0
- Графики показывают пустую область с надписью «Нет данных за период»

Случай 2: resolved_rate = 0% (все алерты open)
- KPI-карточка: красная, значение «0%»
- Не скрывать — это важный сигнал

Случай 3: HIGH-алертов = 0 за период
- KPI-карточка: зелёная (ниже нормы = хорошо)

Случай 4: Chart.js получил пустой dataset
- График отрисовывается с пустыми осями
- Нет JavaScript ошибки

15. Проверочные критерии (Validation)

Спецификация считается выполненной когда:

  • Все 5 экранов реализованы согласно ASCII-вайрфреймам
  • Все REQ-DD с приоритетом MUST реализованы (45 требований)
  • Все 20 AC-DD прошли ручную проверку
  • Polling работает на всех экранах с указанными интервалами
  • SeverityBadge использует точные цвета: HIGH=#ef4444, MEDIUM=#f59e0b, LOW=#10b981
  • Роль viewer не видит кнопок изменения статуса
  • OfflineBanner появляется при потере соединения
  • Автоматический logout при HTTP 401
  • KPI-карточки resolved_rate цветовая индикация работает корректно
  • Trip Details корректно загружает /api/chats/{trip_id}
  • Breadcrumb навигация работает для обоих сценариев (из Trips и из Alerts)
  • Экран Statistics перерисовывает графики при смене периода
