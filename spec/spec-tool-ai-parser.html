Спецификация AI Parser (S0-F02)

Marshall AI Listener — v1.0 | 2026-03-06

1. Цель и скоуп

AI Parser — компонент, который принимает сырые текстовые сообщения из диспетчерских чатов и извлекает из них структурированные логистические данные через LLM (Large Language Model). Парсер преобразует неструктурированный текст в JSON с полями: trip_id, маршрут, статус, заказчик, срочность, проблемы и уровень уверенности.

1.1 Входные данные

  • Raw message: Текст сообщения из чата + metadata (sender_name, chat_name, timestamp, chat_history)
  • Chat history: Опционально — последние N сообщений из чата для контекста (определение trip_id)
  • Customer rules: Справочник правил для 6 заказчиков (Тандер, WB, X5, Магнит, Сельта, Сибур)

1.2 Выходные данные

  • Parsed message JSON: Структурированный результат с полями trip_id, route_from, route_to, slot_time, status, customer, urgency, issue, confidence
  • PostgreSQL запись: Сохранение результата в таблицу parsed_messages
  • Флаг низкого качества: confidence < 0.70 → запись помечается для ручного ревью

1.3 Связь с другими модулями

Chat Listener (S0-F01) ↓ raw_messages (очередь)
AI Parser (S0-F02) ↓ parsed_data (JSON)
Alert Engine (S0-F03) ↓ alerts
Dashboard API
  • ← От Chat Listener: Получает очередь сообщений через RabbitMQ / Redis / File Queue
  • → К Alert Engine: Передаёт parsed_data для анализа правил
  • → К PostgreSQL: Сохраняет результаты в таблицу parsed_messages
  • ← LLM API: Обращается к MiniMax M2.5 (основной), Groq Llama 3.3 70B (fallback)

3. Требования, ограничения и рекомендации

3.1 Функциональные требования (REQ)

REQ-AP-001

Парсинг trip_id с точностью ≥85%

Парсер должен точно распознавать trip_id из сообщений диспетчеров в ≥85% случаев.

Критерии:

  • trip_id может быть в начале сообщения ("Рейс 4521, ..."), в середине, или отсутствовать
  • Если trip_id в контексте (предыдущие сообщения), парсер должен использовать контекст для связи
  • trip_id может быть представлен как "Рейс 4521", "4521", "рейс 4521", "4521-Москва-Краснодар"
  • Ложные распознавания (например, "214.50" как trip_id): не должны быть рассмотрены, confidence < 0.5
REQ-AP-002

Извлечение маршрута (route_from, route_to)

Парсер должен извлекать географические маршруты из сообщения.

Критерии:

  • route_from, route_to: города/регионы (Москва, Краснодар, СПб, Ростов и т.д.)
  • Если маршрут не упоминается явно, поле = null
  • Маршруты могут быть сокращены (МСК-КРД, Москва-Краснодар) — парсер должен нормализовать
  • Опечатки в названиях городов: не должны быть исправлены автоматически (confidence ↓)
REQ-AP-003

Определение статуса рейса

Парсер должен определять текущий статус рейса из сообщения.

Критерии:

  • Допустимые статусы: assigned, loading, in_transit, unloading, completed, problem
  • "Рейс назначен", "Водитель подтверждает" → status = assigned
  • "На месте погрузки", "Погрузка началась" → status = loading
  • "Выехал", "В пути" → status = in_transit
  • "Выгрузка началась" → status = unloading
  • "Прибыл", "Завершен" → status = completed
  • "Проблема", "сломалось", "опаздываю", "ДТП" → status = problem
REQ-AP-004

Извлечение времени слота

Парсер должен извлекать время слота прибытия/выезда на погрузку.

Критерии:

  • slot_time: время в формате HH:MM (14:00, 09:30) или null если не упоминается
  • Примеры: "слот WB 14:00", "выезжаю в 14:00", "слот на погрузку 9:30"
  • Если время упоминается без контекста слота (просто "в 14:00") — попытка использовать контекст
  • Время может быть в диапазоне (14:00-15:00) → брать начало диапазона
REQ-AP-005

Определение заказчика

Парсер должен идентифицировать заказчика из сообщения.

Критерии:

  • Допустимые заказчики: Тандер, WB, X5, Магнит, Сельта, Сибур (+ other)
  • Примеры: "слот WB 14:00" → customer = "WB", "Тандер Ростов" → customer = "Тандер"
  • Если заказчик не упоминается явно, попытка определить из контекста (предыдущие сообщения)
  • Если не удаётся — customer = null
REQ-AP-006

Определение уровня срочности (urgency)

Парсер должен оценивать срочность проблемы.

Критерии:

  • Допустимые уровни: low, medium, high
  • HIGH — поломка оборудования, опоздание >4ч, ДТП, проблема с безопасностью, срыв рейса
  • MEDIUM — опоздание 40 мин - 4ч, простой >1ч, предупреждение, проблема контекстная
  • LOW — информационное сообщение, статус OK, плановое обновление
  • Если status = problem → urgency ≥ medium
REQ-AP-007

Извлечение описания проблемы (issue)

Парсер должен извлекать текст проблемы если она упоминается.

Критерии:

  • issue: строка с описанием проблемы (опоздание, поломка, простой и т.д.) или null если проблемы нет
  • "Стою в пробке на МКАД, опаздываю минут на 40" → issue = "Пробка, опоздание ~40 мин"
  • "реф не выходит на температуру, показывает +8" → issue = "Поломка рефрижератора, температура +8 вместо нормы"
  • "На месте, всё норма" → issue = null
  • Максимум 150 символов
REQ-AP-008

Confidence score для каждого парсинга

Парсер должен возвращать уровень уверенности в результате.

Критерии:

  • confidence: число от 0.0 до 1.0
  • Полная уверенность (confidence ≥ 0.90): ясный trip_id, ясный маршрут, явно указан заказчик
  • Средняя уверенность (0.70-0.89): trip_id определён из контекста, маршрут ясен
  • Низкая уверенность (< 0.70): trip_id отсутствует, маршрут неясен, много неоднозначности
  • confidence = 0.0: парсинг не удался (message не распознана как логистическое сообщение)
REQ-AP-009

Использование контекста чата для определения trip_id

При отсутствии явного trip_id в текущем сообщении, парсер должен обращаться к истории чата.

Критерии:

  • Если trip_id не в текущем сообщении, смотреть последние N сообщений (N = 10) из того же чата
  • Если в истории есть сообщение с trip_id и статусом (assigned/loading), использовать его как контекст
  • Максимум 5 минут истории (не брать очень старые сообщения)
  • Если контекст используется → confidence -0.15
REQ-AP-010

Дедупликация сообщений

Парсер не должен обрабатывать одно и то же сообщение дважды в окне 5 минут.

Критерии:

  • Дедупликация по (message_text, sender_id, timestamp) → hash(text)
  • В памяти или Redis: ключ = message_hash, TTL = 5 мин
  • Если сообщение уже обработано и в кеше → skip, не обращаться к LLM
  • Log: "Message dedup: hash=... skipped"
REQ-AP-011

Fallback chain: MiniMax → Groq → Regex

При сбое основного LLM, парсер должен иметь fallback механизм.

Критерии:

  • Попытка 1: MiniMax M2.5 (основной), timeout = 5 сек
  • Попытка 2: Groq Llama 3.3 70B (если MiniMax ошибка/timeout), timeout = 5 сек
  • Попытка 3: Regex-парсинг (базовое распознавание trip_id, маршрута) если оба LLM упали
  • При fallback на Groq/Regex → confidence -0.25
  • Log каждый fallback: "MiniMax timeout, fallback to Groq"
REQ-AP-012

Rate limiting для API LLM

Парсер должен ограничивать частоту запросов к LLM API.

Критерии:

  • MiniMax: ≤ 100 запросов/мин (по тарифу)
  • Groq: ≤ 100K токенов/день
  • Очередь сообщений: если парсер отстаёт, сообщения накапливаются в Redis/RabbitMQ (не теряются)
  • Log: "Rate limit exceeded, message queued"
  • При превышении rate limit → откладывать парсинг, не отбрасывать

3.2 Нефункциональные требования (CON)

CON-AP-001

Латентность парсинга

Время от поступления сообщения до результата парсинга ≤ 5 секунд.

Критерии:

  • Начало: сообщение получено Chat Listener'ом
  • Конец: результат записан в PostgreSQL и передан в Alert Engine
  • Включает: вызов LLM API, обработка результата, запись в БД
  • P95 латентность: ≤ 5 сек; P99: ≤ 10 сек
CON-AP-002

Максимальный объём обработки

Парсер должен обрабатывать ≤ 30 сообщений в секунду.

Критерии:

  • Rate limiting: 30 msg/sec от Chat Listener'а
  • Выше 30 msg/sec → очередь, не потеря данных
  • Монитор: количество сообщений в очереди (Alert: если >1000)
CON-AP-003

Стоимость LLM токенов

Расход токенов ≤ 50K/день для Sprint 0.

Критерии:

  • Бюджет: 50K токенов/день (MiniMax + Groq вместе)
  • Мониторинг: логирование расхода токенов по часам
  • Срыв бюджета: alert администратору, fallback на Regex
CON-AP-004

Безопасность данных

Парсер не должен хранить или логировать сырые сообщения с персональными данными.

Критерии:

  • Логи содержат только: message_hash, trip_id, customer, status (не содержат имён водителей, номеров телефонов)
  • JSON-логирование в structured format
  • Никакие credentials/tokens в логах
CON-AP-005

Устойчивость к ошибкам

Парсер должен продолжать работу при сбоях.

Критерии:

  • Ошибка в парсинге одного сообщения → не влияет на обработку остальных
  • Исключения логируются, сообщение помечается для ревью (confidence=0)
  • Restart: graceful shutdown на SIGTERM, завершение активных операций за ≤ 10 сек
CON-AP-006

Версионирование промпта

Парсер должен поддерживать версионирование LLM-промпта.

Критерии:

  • Каждый промпт имеет версию (v1, v2, v3...)
  • При изменении промпта → новая версия, A/B-тест результатов
  • Переключение версий через .env переменную (PARSER_PROMPT_VERSION=v2)

4. Интерфейсы и контракты данных

4.1 Входной интерфейс от Chat Listener

Канал доставки: Redis QUEUE или RabbitMQ topic raw_messages

Формат сообщения:

{
  "message_id": "msg_uuid_123",
  "text": "Рейс 4521, Москва-Краснодар, слот на погрузку WB 14:00, реф охладить до +2",
  "sender_id": "user_123",
  "sender_name": "Алексей",
  "chat_id": "chat_456",
  "chat_name": "Marshall Test — WB Рейсы",
  "timestamp": "2026-03-06T09:15:00Z",
  "chat_history": [
    {
      "sender_name": "Диспетчер Мария",
      "text": "Сегодня активна группа WB Ростов",
      "timestamp": "2026-03-06T08:00:00Z"
    }
  ]
}

4.2 Выходной интерфейс в Alert Engine и PostgreSQL

Канал доставки: Redis QUEUE или RabbitMQ topic parsed_messages + прямая запись в DB

Формат результата:

{
  "message_id": "msg_uuid_123",
  "trip_id": "4521",
  "route_from": "Москва",
  "route_to": "Краснодар",
  "slot_time": "14:00",
  "status": "assigned",
  "customer": "WB",
  "urgency": "low",
  "issue": null,
  "confidence": 0.95,
  "parsed_at": "2026-03-06T09:15:01Z",
  "llm_provider": "minimax",
  "prompt_version": "v1"
}

4.3 Структура таблицы PostgreSQL parsed_messages

CREATE TABLE parsed_messages (
  id SERIAL PRIMARY KEY,
  message_id UUID NOT NULL UNIQUE,
  chat_id BIGINT NOT NULL,
  chat_name VARCHAR(255),
  sender_id BIGINT,
  sender_name VARCHAR(255),
  original_text TEXT NOT NULL,

  -- Parsed fields
  trip_id VARCHAR(50),
  route_from VARCHAR(100),
  route_to VARCHAR(100),
  slot_time TIME,
  status VARCHAR(50),
  customer VARCHAR(50),
  urgency VARCHAR(20),
  issue TEXT,
  confidence DECIMAL(3, 2),

  -- Metadata
  parsed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  llm_provider VARCHAR(20),
  prompt_version VARCHAR(10),
  is_deduped BOOLEAN DEFAULT FALSE,
  needs_review BOOLEAN DEFAULT FALSE,

  FOREIGN KEY (chat_id) REFERENCES chats(chat_id),
  INDEX idx_trip_id (trip_id),
  INDEX idx_customer (customer),
  INDEX idx_parsed_at (parsed_at),
  INDEX idx_confidence (confidence)
);

4.4 LLM Промпт (полный шаблон на русском)

Это основной промпт, отправляемый в MiniMax M2.5 и Groq при обработке каждого сообщения.

=== ПРОМПТ ДЛЯ ПАРСЕРА ЛОГИСТИЧЕСКИХ СООБЩЕНИЙ === Ты помощник для парсинга диспетчерских сообщений в логистике. Твоя задача — извлечь структурированные данные из сырого текста сообщений о рейсах. ## ВХОДНЫЕ ДАННЫЕ Сообщение: "{message_text}" Отправитель: {sender_name} Чат: {chat_name} Время: {timestamp} Контекст предыдущих сообщений (если есть): {chat_history} ## СПРАВОЧНИК ЗАКАЗЧИКОВ И ПРАВИЛА 1. Тандер (Yandex Go): - Требование: предохлаждение рефрижератора до +2..+4°C - Обязательно: термописец, медицинская книжка - Штраф за нарушение температуры, опоздание на слот - Триггер HIGH urgency: температура не в диапазоне, опоздание >30 мин 2. WB (Wildberries): - Требование: регистрация в WB Drive - Строгие временные слоты (±15 мин) - Штраф за опоздание на слот (начинается с 15% за каждые 30 мин) - Триггер HIGH urgency: опоздание >30 мин на слот 3. X5 Retail Group: - Требование: НЕЛЬЗЯ стоять на территории грузоотправителя (только на внешней парковке) - Штраф за несанкционированный простой на территории - Триггер HIGH urgency: простой на территории >30 мин 4. Магнит: - Требование: МОЖНО стоять на территории (отличие от X5!) - Менее жёсткие слоты - Триггер MEDIUM urgency: опоздание >1 часа 5. Сельта: - Требование: техника безопасности (каска, жилет, ботинки с железным носком) - Штраф за нарушение ОТ - Триггер HIGH urgency: упоминание о нарушении техники безопасности 6. Сибур: - Требование: особые условия безопасности (опасные грузы) - Свои правила в зависимости от груза - Триггер HIGH urgency: упоминание о проблеме с безопасностью ## ИЗВЛЕКАЕМЫЕ ПОЛЯ Верни JSON с точно этой структурой: { "trip_id": "<строка или null>", "route_from": "<город или null>", "route_to": "<город или null>", "slot_time": "<HH:MM или null>", "status": "<assigned|loading|in_transit|unloading|completed|problem или null>", "customer": "<Тандер|WB|X5|Магнит|Сельта|Сибур|other или null>", "urgency": "<low|medium|high>", "issue": "<строка до 150 символов или null>", "confidence": <число от 0.0 до 1.0> } ## ПРАВИЛА ПАРСИНГА ### trip_id - Ищи в начале сообщения ("Рейс 4521", "4521,", "Рейс №4521") - Может быть число (4521) или строка с буквами (ТР-4521) - Если явно нет, но в контексте предыдущего сообщения есть — используй контекст - Не парси цены, координаты, другие числа как trip_id - Если неуверен — confidence ↓ ### route_from и route_to - Ищи названия городов (Москва, Краснодар, Ростов, СПб, Казань и т.д.) - Формат может быть "Москва-Краснодар", "из Москвы в Краснодар", "МСК → КРД" - Если маршрут не упоминается явно — поля = null - Опечатки не исправляй (это снижает confidence) ### slot_time - Ищи время в формате HH:MM (14:00, 09:30, 23:59) - Контекст: "слот", "выезжаю", "буду в", "прибуду", "погрузка начнётся" - Если диапазон (14:00-15:00) — возьми начало - Время без контекста — если очень вероятно слот, то берём ### status (важно!) - "Рейс назначен", "Водитель подтверждает" → assigned - "На месте", "Приехал на погрузку", "На складе" → loading - "Выехал", "В пути", "Выезжаю", "На дороге" → in_transit - "Выгрузка", "Разгружаем" → unloading - "Прибыл", "Завершен", "Рейс закрыт" → completed - "Проблема", "Сломалось", "Опаздываю", "ДТП", "Простой" → problem ### customer - Явное упоминание: "WB", "Тандер", "X5", "Магнит", "Сельта", "Сибур" - Может быть аббревиатура: "ВБ" = WB, "Х5" = X5 - Если в контексте упоминался — попытка использовать контекст ### urgency (логика) - HIGH: status=problem + (поломка ИЛИ опоздание >4ч ИЛИ ДТП ИЛИ ОТ-нарушение ИЛИ срыв слота WB/Тандер >30мин) - MEDIUM: status=problem + (опоздание 40 мин - 4ч ИЛИ простой >1ч ИЛИ предупреждение) - LOW: status != problem ИЛИ информационное сообщение ### issue - Краткое описание проблемы, если есть (≤150 символов) - Примеры: - "Пробка на МКАД, опоздание ~40 мин" - "Поломка рефрижератора, температура +8 вместо +3" - "Простой на X5 60 мин, не разгружают" - null если проблемы нет ### confidence - 0.95–1.0: trip_id явный, маршрут ясен, заказчик явный, статус ясен - 0.85–0.94: trip_id явный, маршрут ясен, заказчик определён из контекста - 0.70–0.84: trip_id из контекста, есть неоднозначность в маршруте - 0.50–0.69: очень неуверен, много пропусков данных - < 0.50: сообщение может вообще не быть о логистике ## ФИНАЛЬНЫЕ ИНСТРУКЦИИ 1. Верни ТОЛЬКО JSON, без дополнительного текста 2. Все поля обязательны 3. Если поле неизвестно — используй null (не пустую строку) 4. Confidence всегда число от 0.0 до 1.0 5. urgency всегда одно из: low, medium, high (по умолчанию = low если status != problem) 6. Дважды проверь trip_id и customer, это критично для Alert Engine Приступай к парсингу!

Случай 2: Сообщение с опечатками и неправильной грамматикой

Обоснование: trip_id, customer ясны, но маршрут с опечаткой, время невалидно → confidence ниже.

"Выехал на погрузку"

(тот же текст, тот же sender)

→ Дедупликация срабатывает, сообщение НЕ парсится повторно

Логирование: "Message dedup: hash=abc123... skipped (5min window)"

8.3 Performance-граничные случаи

Случай 1: Пик нагрузки — 60 сообщений за 30 сек

Сценарий: Диспетчер создал сообщения в группе, все приходят пачкой.

Ожидание:

  • Первые 30 msg (в течение первых 30 сек) обрабатываются сразу
  • Оставшиеся 30 встают в очередь Redis
  • Очередь обрабатывается со скоростью 30/сек (после разгрузки)
  • Максимальная задержка: <2 мин
  • Все сообщения в итоге обработаны

Метрики:

Queue depth: 30 (max)
Processing time: ~40 sec total
Rate limit violations: 0
Messages lost: 0

Случай 2: Недостаточно бюджета токенов (>50K/день)

Сценарий: День идёт к концу, уже использовано 48K токенов из 50K.

Поступает новое сообщение:

  • MiniMax может потребовать ~500 токенов
  • Остаток: 50K - 48K = 2K токенов

Ожидание:

  • Парсер видит что будет превышение
  • Fallback на Groq (бесплатные 100K токенов/день имеют отдельную квоту)
  • Groq обрабатывает сообщение
  • Log alert: "Token budget MiniMax exceeded, using Groq"
