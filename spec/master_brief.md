## 2. Описание проблемы

Диспетчеры координируют рейсы через 4+ мессенджера одновременно (WhatsApp, Telegram, Viber, Max). Ключевые боли:

1. **Хаос в каналах связи** — сообщения теряются между платформами, водитель не знает кто на смене
2. **Нет контроля статусов** — прибытие, транзитное время, простой — всё на словах, нет записей
3. **Документы теряются** — термограммы, фотоотчёты, ТСД забываются или задерживаются
4. **Штрафы до 100%** — опоздание >4ч = 15%, >6ч = 30% (= срыв рейса), >24ч = 100% (груз считается утраченным). Средний рейс = 80 000 руб, один штраф 30% = 24 000 руб потерь

## 3. Решение: AI Listener (Sprint 0)

**Пассивная AI-система**, которая подключается к существующим групповым чатам диспетчеров и:
- Слушает ВСЕ текстовые сообщения (нулевые изменения в процессах людей)
- Парсит структурированные логистические данные через LLM (trip_id, маршрут, статус, заказчик, urgency)
- Отображает алерты о критических проблемах в дашборде
- Предоставляет аналитический дашборд с ключевыми метриками

**Ключевой принцип:** Ноль изменений для пользователей. Люди работают как раньше. AI просто наблюдает и структурирует.

### Что НЕ входит в Sprint 0 (отложено на платные спринты):
- Голос в текст (Whisper STT) — будущий спринт
- Интерактивный бот для водителей — Sprint 1
- Чек-листы, кнопки статусов — Sprint 1
- Контроль документов — Sprint 1
- Мультиплатформа (Max, WhatsApp) — Sprint 1+

## 6. Варианты подключения к чатам

Оценены три подхода. Рекомендация: реализовать как pluggable-адаптер.

| Вариант | Подход | Лучше для | Риск |
|---------|--------|----------|------|
| A: MTProto | Telethon user account (StringSession) | Быстрый пилот, чтение истории | Риск бана аккаунта |
| B: Bot API | Бот добавляется в групповые чаты | Продакшн, Marshall контролирует | Нужно добавить бота в каждый чат, нет истории |
| C: Пересылка | Автопересылка в канал-коллектор | Резерв, если A/B невозможны | Теряется контекст отправителя |

**Рекомендация:** Bot API (B) для продакшна + MTProto (A) для пилота. Переключатель в конфиге: `LISTENER_MODE=bot|mtproto`.

## 13. Ключевые решения

1. **Только пассивное прослушивание** — никаких ответов бота в Sprint 0
2. **PostgreSQL** вместо SQLite — масштаб для 500+ партнёров
3. **MiniMax M2.5** как основной LLM — уже проверен в авто-респондере
4. **Нет голоса/STT** в Sprint 0 — отложено
5. **Pluggable-адаптер** для подключения к чатам (MTProto vs Bot API)
6. **FastAPI + Chart.js** для дашборда — лёгкий, без фреймворка
7. **Docker Compose** для деплоя
8. **Нет алерт-бота** в Sprint 0 — алерты только в дашборде (лента/список)
9. **Деплой на RUVDS Тима** — Marshall не имеет доступа к серверу
10. **Нет интеграции с АРМ/ТМС** — Sprint 0 автономен
11. **DM включены** — listener может захватывать и директные сообщения диспетчерам
