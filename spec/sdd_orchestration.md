# План оркестрации SDD — Marshall AI Listener

## Архитектура

```
ОРКЕСТРАТОР (Claude Opus, основная сессия)
    |
    |-- готовит контекстные брифы
    |-- запускает агентов
    |-- ревьюит результат
    |-- разрешает конфликты между спеками
    |-- интегрирует финальную архитектуру
    |
    +-- АГЕНТЫ (специалисты, в песочнице)
         |-- product-manager           — скоуп, метрики, JTBD
         |-- specification (x5)        — пишет spec-файлы
         |-- database-architect        — схема PostgreSQL
         |-- api-designer              — контракты REST API
         |-- se-ux-ui-designer         — UX-флоу, JTBD экранов
         |-- ascii-ui-mockup-generator — вайрфреймы
         |-- ui-designer               — визуальный дизайн
```

## Пайплайн спринтов

### Спринт 1: СБОРКА КОНТЕКСТА [Оркестратор]
- **Вход:** Все существующие материалы (spec-v1.yaml, architecture.html, база знаний, презентация)
- **Выход:** `spec/master_brief.md`
- **Статус:** ГОТОВО

### Спринт 2: СКОУП ПРОДУКТА [product-manager]
- **Вход:** master_brief.md
- **Выход:** `spec/product_brief.md`
- **Бриф агенту:** Определить MVP-скоуп Sprint 0. Валидировать 5-7 ключевых метрик для дашборда. JTBD для каждой роли. Что убрать, что оставить. Stack rank приоритетов.
- **Зависимости:** Спринт 1

### Спринт 3: ПАРАЛЛЕЛЬНЫЕ СПЕКИ [4 агента одновременно]
- **Вход:** master_brief.md + product_brief.md + модульный контекст
- **Выход:** 4 spec-файла
- **Зависимости:** Спринт 2

| Агент | Входной контекст | Выход | Параллелизуем |
|-------|-----------------|-------|---------------|
| specification | Код авто-респондера + варианты подключения | spec-tool-chat-listener.md | Да |
| specification | LLM-пайплайн + шаблон промпта из spec-v1 | spec-tool-ai-parser.md | Да |
| specification | Правила алертов + эскалация из spec-v1 | spec-tool-alert-engine.md | Да |
| database-architect | Все модели данных из spec-v1 + architecture.html | spec-data-schema.md | Да |

### Спринт 4: API-КОНТРАКТ [api-designer]
- **Вход:** master_brief + схема БД + интерфейсы модулей из Спринта 3
- **Выход:** `spec-tool-dashboard-api.md`
- **Бриф агенту:** Спроектировать REST API для дашборда. Эндпоинты: /api/trips, /api/alerts, /api/stats, /api/chats. В стиле OpenAPI. Авторизация, пагинация, фильтрация.
- **Зависимости:** Спринт 3 (нужна схема + интерфейсы модулей)

### Спринт 5: ФРОНТЕНД [3 агента последовательно]
- **Вход:** master_brief + product_brief + API-спека + список метрик
- **Выход:** `spec-design-dashboard.md` (UX + UI + вайрфреймы в одном документе)
- **Зависимости:** Спринт 2 (метрики) + Спринт 4 (API)

| Шаг | Агент | Задача | Вход от |
|-----|-------|--------|---------|
| 5а | se-ux-ui-designer | JTBD для каждого экрана, пользовательские флоу | product_brief |
| 5б | ascii-ui-mockup-generator | Текстовые вайрфреймы дашборда | выход 5а |
| 5в | specification | Формальная спека с REQ/AC | выход 5а + 5б |

### Спринт 6: ИНТЕГРАЦИОННЫЙ РЕВЬЮ [Оркестратор]
- **Вход:** Все 8 spec-файлов
- **Выход:** `spec-architecture-overview.md` (ADR + системная диаграмма + карта интерфейсов)
- **Задача:** Сверить интерфейсы между модулями. Проверить что контракты данных совпадают. Устранить противоречия. Создать финальный документ архитектуры с Mermaid-диаграммой.
- **Зависимости:** Спринты 3, 4, 5

### Спринт 7: ПЛАН РЕАЛИЗАЦИИ [Оркестратор + product-manager]
- **Вход:** Все спеки + обзор архитектуры
- **Выход:** `spec-process-sprint0-plan.md`
- **Задача:** Разбить на задачи реализации. Оценить часы по модулям. Определить зависимости. Создать порядок выполнения. Привязать к таймлайну доставки Sprint 0.
- **Зависимости:** Спринт 6

## Шаблон брифа для агента

Каждый агент получает:
```
1. master_brief.md (полностью, без изменений)
2. Модульный бриф:
   - ЧТО: что делает этот модуль, его границы
   - ВХОДЫ: какие данные/события получает
   - ВЫХОДЫ: какие данные/события производит
   - ИНТЕРФЕЙСЫ: как связан с другими модулями
   - СУЩЕСТВУЮЩИЙ КОД: релевантные фрагменты из авто-респондера или spec-v1
   - ОГРАНИЧЕНИЯ: что НЕ включать (границы скоупа)
3. Шаблон спецификации (от агента specification)
4. Инструкция: "Напиши полную спецификацию по шаблону.
   Фокус на требованиях, критериях приёмки и интерфейсах.
   НЕ пиши код реализации."
```

## Критерии качества

Каждая спека проходит проверку перед следующим спринтом:
1. **Все REQ-xxx имеют соответствующие AC-xxx** (нет требования без критерия приёмки)
2. **Интерфейсы документированы** (типы входов/выходов, обработка ошибок)
3. **Нет расползания скоупа** (ничего за пределами MVP Sprint 0)
4. **Согласованность с master_brief** (тех. стек, ограничения, решения)
5. **Самодостаточность** (другой разработчик может реализовать только по этой спеке)
