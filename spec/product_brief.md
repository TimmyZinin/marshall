# Marshall AI Listener — Продуктовый бриф (Sprint 0)

## 1. Видение продукта

**Пассивная AI-система, которая слушает диспетчерские чаты, структурирует данные через LLM и показывает менеджерам критические события в реальном времени — для снижения штрафов и хаоса в логистике.**

---

## 2. Jobs-to-be-Done (JTBD)

### 2.1 Диспетчер (косвенный пользователь)
**Когда** я координирую рейсы через 4+ мессенджера одновременно,
**я хочу** продолжать работать как обычно (без изменения процесса),
**чтобы** система сама парсила мои сообщения и помогала менеджеру видеть проблемы.

### 2.2 Менеджер (прямой пользователь дашборда)
**Когда** случается проблема с рейсом (опоздание, поломка, простой),
**я хочу** увидеть её в дашборде за секунды, с деталями и статусом,
**чтобы** оперативно связаться с диспетчером и избежать штрафа.

### 2.3 Руководство Marshall (прямой пользователь дашборда)
**Когда** заканчивается смена или рабочий день,
**я хочу** посмотреть аналитику: сколько рейсов, сколько проблем, какие штрафы,
**чтобы** оценить операционную эффективность и улучшать процессы.

## 3. MVP-скоуп Sprint 0

Sprint 0 — **БЕСПЛАТНЫЙ пилот**, цель: показать ценность, зацепить клиента для платных спринтов ($250/час далее).

### S0-F01: Chat Listener — Подключение и чтение

**Описание:**
Система подключается к 3 тестовым группам Telegram диспетчеров, читает все новые текстовые сообщения, включая прямые сообщения (DM). Сохраняет raw-сообщения в БД.

**Ключевые сценарии:**
- Диспетчер напишет в групповой чат — Listener захватит сообщение <100 мс
- Водитель отправит DM диспетчеру — Listener распарсит из истории + новых сообщений
- Сообщение содержит текст (нет файлов) — парсится и сохраняется

**Входит в скоуп:**
- Подключение через Telethon (MTProto) как pluggable-адаптер
- Чтение из 3 заранее выбранных групповых чатов + DM
- Асинхронный event listener на asyncio
- Graceful shutdown + логирование структурированное (JSON)
- Rate limiting 30 сообщений/сек (не перегружать LLM)

**Выходит из скоупа:**
- Bot API (отложено на Sprint 1, требует добавления бота в чаты)
- Голос/Whisper STT (отложено на Sprint 1)
- Max, WhatsApp, Viber (отложено на Sprint 1+)
- История сообщений старше 30 дней (дорого, не нужно для пилота)

### S0-F02: AI Parser — LLM-парсинг структурированных данных

**Описание:**
Каждое новое сообщение отправляется в LLM (MiniMax M2.5, Groq fallback), парсер извлекает структурированные поля: trip_id, маршрут, статус, заказчик, urgency, проблемы. Сохраняет результат в БД.

**Ключевые сценарии:**
- "Рейс 4521, Москва-Краснодар, слот WB 14:00" → trip_id=4521, route=Moskva-Krasnodar, status=assigned
- "Стою в пробке, опаздываю минут на 40" → urgency=medium, issue="Traffic, 40min delay"
- "Реф не выходит на температуру +8" → urgency=high, issue="Refrigerator malfunction, temp +8 vs norm"

**Входит в скоуп:**
- Шаблонизированный промпт для парсинга (5-слойный пайплайн из авто-респондера)
- Извлечение полей: trip_id, route_from, route_to, slot_time, status, customer, urgency, issue, confidence
- Fallback: MiniMax → Groq (100K бесплатных токенов/день)
- Кеширование промпта (не парсить дубли в окне 5 мин)
- Обработка ошибок: парсер не смог → запись с confidence=0, флаг для ревью

**Выходит из скоупа:**
- STT (голос в текст) — отложено
- Распознавание фото/документов — отложено
- Парсинг максимум из "очень неструктурированного текста" (требует fine-tuning) — базовый парсинг Llama-level

### S0-F03: Alert Engine — Детекция критических событий

**Описание:**
На основе parsed-данных система определяет критические ситуации (опоздание >4ч, поломка, простой >2ч) и выставляет алерты в дашборде. Никаких уведомлений в Telegram, только дашборд.

**Ключевые сценарии:**
- Водитель сообщил "опаздываю 40 мин" при слоте 14:00 → Alert: "Possible delay 40min on trip 4521 WB"
- Водитель: "Реф сломалась, температура +8" → Alert: "CRITICAL: Refrigerator down on trip 7803, temp +8"
- Водитель стоит на месте >2 часов → Alert (если X5) или OK (если Магнит, они разрешают)

**Входит в скоуп:**
- Правила для 6 заказчиков: Тандер (предохлаждение +2..+4), WB (слоты), X5 (не стоять), Магнит (можно стоять), Сельта (техника безопасности), Сибур (спецусловия)
- 5 типов алертов: delay, equipment_failure, downtime, safety_violation, docs_missing
- Severity: HIGH (блокирует рейс), MEDIUM (требует внимания), LOW (информационный)
- Эскалация: Alert создан → можно отметить как reviewed/resolved в дашборде
- Нет отправки в Telegram (только дашборд показывает)

**Выходит из скоупа:**
- Отправка алертов в Telegram/бот — отложено на Sprint 1
- Автоматическая эскалация (перевод на диспетчера смены) — отложено на Sprint 1
- SMS/push-уведомления — отложено

### S0-F04: Dashboard — Веб-дашборд для просмотра

**Описание:**
Одностраничный веб-дашборд (HTML + Chart.js) для менеджеров. Две вкладки: "Активные рейсы" (таблица) и "Алерты" (лента). Real-time обновление данных через REST API.

**Ключевые сценарии:**
- Менеджер открывает дашборд → видит 12 активных рейсов (trip_id, маршрут, статус, заказчик, время погрузки)
- 2 алерта выделены красным (HIGH) — опоздание WB и поломка рефа
- Нажимает на алерт → вскрывается история сообщений по этому рейсу
- Фильтр по заказчику (Тандер/WB/X5/Магнит) — видит только релевантные рейсы
- Граф за день: сколько рейсов завершено, средняя задержка, количество алертов

**Входит в скоуп:**
- REST API (FastAPI): /api/trips, /api/alerts, /api/stats, /api/chats/{trip_id}
- Базовая авторизация (пароль в .env, не OAuth)
- Две вкладки: список активных рейсов + лента алертов
- Чарты: 3 графика (всего рейсов, среднее время в пути, количество HIGH-алертов за день)
- Фильтры: по заказчику, по дате, по статусу (active/completed)
- Обновление данных каждые 3 сек (WebSocket или polling)

**Выходит из скоупа:**
- Мобильная версия (только веб)
- Экспорт в Excel
- Кастомные дашборды, drag-and-drop
- Интеграция с АРМ Marshall — отложено на платные спринты
- Авторизация через OAuth/SSO

## 4. Метрики успеха Sprint 0

| № | Метрика | Целевое значение | Как измерять | Критичность |
|---|---------|-----------------|-------------|------------|
| М1 | Точность парсинга trip_id | ≥85% | Вручную проверить 100 сообщений из 3 чатов, сколько trip_id распознан правильно | HIGH |
| М2 | Latency (сообщение → дашборд) | ≤5 сек | Засечь время: сообщение в чат → появилось на дашборде | HIGH |
| М3 | Uptime системы | ≥99% за 7 дней | Мониторить: Container up, PostgreSQL up, API respond 200 | HIGH |
| М4 | Количество обнаруженных HIGH-алертов за день (3 чата) | 5–15 за день | Счётчик в БД: SELECT COUNT(*) WHERE severity='HIGH' | MEDIUM |
| М5 | LLM токены/день (реальный расход) | ≤50K/день | Логи API MiniMax + Groq | MEDIUM |
| М6 | Покрытие заказчиков в Alert Engine | 100% (6 заказчиков) | Проверить: правила для Тандер, WB, X5, Магнит, Сельта, Сибур загружены | HIGH |
| М7 | Успешность: "Николя видит алерт и понимает что делать" | ≥90% | Юзер-тест: Николя открывает дашборд с 5 реальными сценариями, правильно ли он понял что произошло | HIGH |

## 5. Приоритеты (Stack Rank) — RICE-скоринг

| Ранг | Фича | Reach | Impact | Confidence | Effort | RICE | Статус |
|------|------|-------|--------|------------|--------|------|--------|
| 1 | S0-F02: AI Parser | 12 рейсов/день | High (без парсинга — нет алертов) | High (LLM работает) | 40h | 900 | **MUST** |
| 2 | S0-F03: Alert Engine | 12 рейсов/день | High (основная ценность) | High (правила ясны) | 30h | 1200 | **MUST** |
| 3 | S0-F01: Chat Listener | 12 рейсов/день | High (без Listener — нет данных) | Medium (MTProto риски) | 20h | 400 | **MUST** |
| 4 | S0-F04: Dashboard | 2 менеджера | High (видимость = решение проблем) | High (простой фронтенд) | 24h | 1200 | **MUST** |

**Обоснование порядка:**
1. **Parser** — критическая зависимость для Alert Engine, наибольший RICE (complex LLM, biggest impact)
2. **Alert Engine** — второй по RICE, преобразует parsed данные в бизнес-ценность (опоздания → штрафы)
3. **Listener** — третий: нужен для тестирования, но в демо можно использовать mock-данные (снижает RICE через Effort)
4. **Dashboard** — четвёртый: нужен для show-off, но технически проще других (простой фронтенд + REST API)

**Порядок реализации (спринт):**
1. Listener (недель 1–2, чтобы захватить реальные данные для тестирования Parser)
2. Параллельно: Parser (недель 1–3) + Alert Engine (недель 2–3)
3. Dashboard (недели 3–4, после фиксации API)

## 6. Что выводим из скоупа (Out of Scope)

**Явный список того, что НЕ делаем в Sprint 0:**

- **Голос и Whisper STT** — распознавание голосовых сообщений отложено на Sprint 1
- **Интерактивный бот с ответами** — водители не получают автоответы, только читаем их сообщения
- **Кнопки статусов** ("На месте", "Выгрузка начата") — отложено на Sprint 1
- **Мультиплатформа** (Max, WhatsApp, Viber) — начинаем только с Telegram, Max ≈апрель 2026 при блокировке
- **Интеграция с АРМ/ТМС Marshall** — автономная система, не читает API Marshall
- **Контроль документов** (фото ТСД, чеки Почты) — только текст из сообщений
- **МОВИЗОР интеграция** — GPS-трекинг не подключаем, только текстовые сообщения от водителей
- **Алерты в Telegram** — только дашборд (решение на Sprint 1, требует бота)
- **Расписание смен** — все данные видны всем с доступом к дашборду, не учитываем смены диспетчеров
- **SMS/Email уведомления** — отложено
- **Авторизация OAuth/LDAP** — простой пароль в .env (для пилота)
- **Экспорт/аналитика Excel** — только веб-дашборд
- **Мобильное приложение** — только веб
- **Кеширование истории >30 дней** — дорого, не нужно для пилота

## 7. Риски и допущения

| Риск | Вероятность | Влияние | Инструмент |
|------|-------------|--------|-----------|
| **Блокировка Telegram с 1 апреля 2026** | High | Критическое: система не сможет читать чаты | Подготовить адаптер для Max (VK API), готов к Switch за <7 дней |
| **Качество парсинга на реальных сообщениях** | Medium | High: если точность <70%, алерты будут ложные | User-test с 100 реальными сообщениями, итерировать промпт |
| **Объём сообщений >30/сек (rate limiting)** | Low | Medium: потеряем часть данных | Rate limit 30/сек, очередь на диске, логирование dropped msgs |
| **Аккаунт Telegram забанится** | Low | High: система не сможет читать | Использовать сервисный аккаунт, не личный диспетчера |
| **API MiniMax неработает в момент Х** | Medium | Medium: падаем на Groq (100K/день бесплатно) | Fallback-цепочка: MiniMax → Groq → парсинг на симптомах (regex) |
| **Николя скажет "это бесполезно, я и сам всё вижу"** | Medium | Critical: невозможно продать Sprint 1 | Демо на реальных сценариях, показать как система сэкономила бы $XX в штрафах |
| **Клиент попросит "а давайте интеграцию с АРМ сейчас"** | High | High: расширение скоупа, delay | Договорённость заранее: Sprint 0 автономен, Sprint 2+ платное |

**Допущения:**
- 3 тестовых чата будут активны минимум до конца марта 2026
- Структура сообщений от диспетчеров достаточно регулярна (trip_id в начале, маршрут ясен)
- Заказчик согласен с отложением Bot API на Sprint 1 (MTProto ОК для пилота)
- Телефонный номер для Telethon StringSession будет выделен (не личный)
- RUVDS остаётся доступен и не падает (99.5% uptime гарантия)

## 8. Критерии успеха Sprint 0

**Sprint 0 считается успешным, когда:**

1. ✅ **Система работает автономно** — 7 дней без критических ошибок (uptime ≥99%), логируется в JSON
2. ✅ **Parser парсит точно** — ≥85% trip_id и маршрутов распознаны, проверено вручную на 100 сообщениях
3. ✅ **Alert Engine обнаруживает проблемы** — при опоздании водителя на >40 мин система создаёт алерт за <5 сек
4. ✅ **Dashboard работает** — менеджер открывает браузер, видит 12 рейсов, 2 HIGH-алерта, может фильтровать по заказчику
5. ✅ **Демо Николе впечатляет** — показываем 5 сценариев (опоздание, поломка, простой, ДТП, срыв), Николя говорит "wow, это сэкономило бы нам $XXXX в штрафах"
6. ✅ **Код готов к реализации** — полная спецификация 4 модулей (Listener, Parser, Alert, API/Dashboard), тесты, архитектура

**Демо для Николы — 5 сценариев:**
- **Сценарий 1: Опоздание на слот WB** → Alert за 40 мин до слота ("Possible delay on WB trip 4521")
- **Сценарий 2: Поломка рефа (HIGH)** → Alert "Refrigerator down, temp +8"
- **Сценарий 3: Простой на X5 (ошибка)** → Alert "Downtime on X5 (not allowed)" ← система знает правила
- **Сценарий 4: ДТП на трассе (информационный)** → Low-alert "Traffic incident on М4, delays expected"
- **Сценарий 5: Успешный рейс** → GREEN статус, no alerts, рейс завершён

## 9. Путь к монетизации

### 9.1 Sprint 0 (бесплатный пилот) — зацепить клиента

**Что впечатлит Николу:**
- Видит в дашборде ВСЕ проблемы в реальном времени (что скрыто в мессенджерах)
- Получает структурированные данные, а не текстовый хаос
- Понимает: "Если бы я это видел раньше, я бы сэкономил 30% на штрафах"
- Чувствует: система готова к масштабированию (архитектура 500+ партнёров, PostgreSQL, Docker)

**Что мы демонстрируем:**
1. Три рабочих чата (WB, Тандер, Общая диспетчерская)
2. Дашборд с реальными рейсами и алертами
3. Граф: "За 7 дней обнаружено 45 потенциальных штрафов"
4. ROI-расчёт: "Если бы вы действовали на эти алерты, сэкономили бы ~$15K за неделю"

### 9.2 Sprint 1 (платный) — $250/час

**Что Николе нужно дальше:**

| Фича | Проблема Sprint 0 | Решение Sprint 1 | Ценность |
|------|------------------|-----------------|---------|
| **Алерты в Telegram** | Менеджер должен постоянно смотреть дашборд | Bot отправляет HIGH-алерты мгновенно | Не упустит критическую проблему |
| **Bot API** (не MTProto) | Риск бана, Marshall не контролирует | Bot добавляется в каждый чат, Marshall контролирует | Production-ready, без рисков |
| **STT (голос)** | Диспетчер говорит по телефону, а Listener не слышит | Whisper интегрирован в listener | Покрывает 100% коммуникации |
| **Контроль документов** | Теряются фото ТСД, чеки | Bot запрашивает фото + чек после выгрузки | Никакие документы не потеряются |
| **МОВИЗОР интеграция** | Нет GPS, только текст | Подключаем GPS из МОВИЗОР, соединяем с парсингом | Видим реальную геолокацию, не полагаемся на слова |
| **АРМ интеграция** | Алерты не синхронизируются с AРМ Marshall | API читает trip_id → подтягивает все данные из АРМ | Single source of truth |
| **Max (VK) адаптер** | Telegram блокируется 1 апреля | Max подключается за неделю (pluggable) | Готовы к переходу |

**Коммерческая логика:**
- Sprint 0: показываем ценность за $0 → дешево, быстро, впечатляет
- Sprint 1: покупают за $250/ч для "production-ready" (Bot API, алерты в TG, STT)
- Sprint 1 = 40–60 часов = $10K–15K (в месяц помесячно в зависимости от развития)
- Upgrade path: Listener → Parser → Alerts → Bot → STT → Docs → МОВИЗОР → АРМ

### 9.3 Долгосрочная стратегия (Sprint 2+)

После Sprint 1, когда система работает, есть два пути:

1. **SaaS (Recurring Revenue)**: $500–1000/мес за Marshall всей компании (включает все спринты)
2. **Implementation Services**: $250/ч для интеграции, обучения, кастомизации (как сейчас)
