# Marshall AI Listener — Спецификация дизайна дашборда

Данный документ определяет полный дизайн веб-дашборда системы Marshall AI Listener: пользовательский опыт, вайрфреймы всех экранов, формальные требования к интерфейсу и компонентную модель. Документ является руководством для разработки фронтенда в рамках Sprint 0.

---

## 1. Назначение и область применения

### 1.1 Цель документа

Данная спецификация определяет:
- Пользовательские сценарии (JTBD) для каждого из 5 экранов дашборда
- Информационную архитектуру и иерархию элементов интерфейса
- ASCII-вайрфреймы всех экранов
- Формальные требования (REQ-DD) и критерии приёмки (AC-DD)
- Компонентную модель, цветовую систему и типографику

### 1.2 Аудитория документа

- **Frontend-разработчик Sprint 0** — основная аудитория, реализует интерфейс
- **Тим Зинин (Zinin Corp)** — владелец продукта, принимает дизайн
- **Николя Бобров (Marshall)** — конечный пользователь уровня руководства

### 1.3 Границы области применения

**В области применения:**
- Веб-дашборд (HTML + vanilla JS + Chart.js)
- 5 экранов: Login, Active Trips, Alerts Feed, Trip Details, Statistics
- 3 роли: admin, manager, viewer
- Polling каждые 3 секунды

**За пределами области применения:**
- Мобильное приложение
- Интерфейс диспетчера (работает в Telegram)
- Административная панель управления LLM-агентом

## 2. Определения

| Термин | Определение |
|--------|-------------|
| JTBD | Jobs To Be Done — методология описания задач пользователя через формат "Когда [ситуация], я хочу [мотив], чтобы [результат]" |
| Рейс (Trip) | Единица грузоперевозки: маршрут из точки А в точку Б, привязанный к конкретному заказчику и водителю |
| Алерт (Alert) | Автоматически сгенерированное AI-системой уведомление о проблемной ситуации в рейсе |
| Severity | Уровень критичности алерта: HIGH / MEDIUM / LOW |
| Trip ID | Уникальный идентификатор рейса, извлекаемый из Telegram-чатов |
| Polling | Периодические HTTP-запросы к API для обновления данных (интервал: 3 сек) |
| JWT | JSON Web Token — механизм авторизации в дашборде |
| Заказчик (Customer) | Клиент Marshall: Тандер, WB, X5, Магнит, Сельта, Сибур |
| Диспетчер | Сотрудник, работающий в Telegram-чатах; НЕ пользователь дашборда |
| Менеджер | Прямой пользователь дашборда; отслеживает активные рейсы и алерты |
| Viewer | Роль только для просмотра (без возможности изменять статусы алертов) |
| KPI | Key Performance Indicator — ключевой показатель эффективности |
| SPA | Single Page Application — одностраничное веб-приложение |
| Chart.js | Библиотека для построения графиков на HTML Canvas |
| Latency | Задержка между событием в чате и появлением данных в дашборде |

## 3. Часть 1: UX-анализ

### 3.1 JTBD для каждого экрана

#### 3.1.1 Экран Login

**Основное JTBD:**
Когда я открываю дашборд утром или после перерыва, я хочу быстро войти под своим логином, чтобы получить доступ только к данным, соответствующим моей роли (менеджер / руководство / admin).

**Дополнительные JTBD:**
- Когда система показывает ошибку входа, я хочу понять причину (неверный пароль / аккаунт заблокирован), чтобы решить проблему самостоятельно или обратиться к admin.
- Когда я нахожусь не у своего компьютера, я хочу чтобы сессия не оставалась активной после закрытия вкладки, чтобы защитить данные.

**Роли и права доступа:**

| Роль | Экраны доступны | Изменение данных |
|------|----------------|-----------------|
| admin | Все 5 экранов | Все операции |
| manager | Active Trips, Alerts Feed, Trip Details, Statistics | PATCH /api/alerts/{id} |
| viewer | Active Trips, Alerts Feed, Trip Details, Statistics | Только чтение |

#### 3.1.2 Экран Active Trips (Активные рейсы)

**Основное JTBD:**
Когда я начинаю рабочий день или возвращаюсь после перерыва, я хочу за 10 секунд увидеть все рейсы в движении и какие из них требуют внимания, чтобы расставить приоритеты на ближайший час.

**Дополнительные JTBD:**
- Когда мне нужно проверить рейсы конкретного заказчика (например, WB), я хочу отфильтровать таблицу по заказчику, чтобы не просматривать все рейсы.
- Когда я вижу рейс с отметкой HIGH-алерта, я хочу кликнуть на него и сразу перейти к деталям, чтобы понять ситуацию.
- Когда статус рейса изменился (например, появилась задержка), я хочу чтобы таблица обновилась автоматически, чтобы не нажимать F5.

#### 3.1.3 Экран Alerts Feed (Лента алертов)

**Основное JTBD:**
Когда поступает новый HIGH-алерт, я хочу сразу увидеть что произошло, в каком рейсе и что нужно сделать, чтобы принять решение в течение 1 минуты.

**Дополнительные JTBD:**
- Когда алерт просмотрен и я связался с водителем, я хочу отметить его как reviewed, чтобы коллеги знали что ситуация взята в работу.
- Когда проблема решена, я хочу отметить алерт как resolved, чтобы он ушёл из активной ленты.
- Когда мне нужно найти все алерты по конкретному рейсу, я хочу отфильтровать ленту по trip_id или заказчику.
- Когда я хочу проверить сколько было алертов за прошлую неделю, я хочу применить фильтр по дате.

#### 3.1.4 Экран Trip Details (Детали рейса)

**Основное JTBD:**
Когда у рейса есть активный алерт, я хочу увидеть полную историю сообщений из чата диспетчеров по этому рейсу, чтобы понять контекст ситуации и принять правильное решение.

**Дополнительные JTBD:**
- Когда я вижу список алертов по рейсу, я хочу видеть их в хронологическом порядке, чтобы понять как развивалась ситуация.
- Когда мне нужно доказательство нарушения (для отчёта), я хочу скопировать конкретное сообщение из истории чата, чтобы включить его в отчёт.

#### 3.1.5 Экран Statistics (Статистика)

**Основное JTBD:**
Когда Николя Бобров подводит итоги дня или недели, он хочет за 30 секунд увидеть: сколько рейсов было, сколько алертов, какой процент решён вовремя, чтобы оценить эффективность работы системы и диспетчеров.

**Дополнительные JTBD:**
- Когда нужно сравнить текущую неделю с прошлой, я хочу переключить период в фильтре, чтобы увидеть динамику.
- Когда я вижу что HIGH-алертов в этот день значительно больше нормы, я хочу провалиться в ленту алертов за этот день, чтобы понять что происходило.

### 3.3 Информационная архитектура

#### 3.3.1 Глобальная навигация

Навигация присутствует на всех экранах кроме Login. Расположена сверху (top navigation bar).

```
Логотип Marshall AI | Active Trips | Alerts Feed | Statistics | [Пользователь: email] [Выйти]
```

Активный пункт меню подсвечивается визуально. Экран Trip Details открывается как дочерний по отношению к Active Trips или Alerts Feed (breadcrumb навигация).

#### 3.3.2 Иерархия экранов

```
Дашборд
├── Login (нет nav)
├── Active Trips (основной экран после входа)
│   └── Trip Details (дочерний экран)
├── Alerts Feed
│   └── Trip Details (дочерний экран, тот же компонент)
└── Statistics
```

#### 3.3.3 Информационная иерархия по экранам

**Login:**
1. Заголовок системы (Marshall AI Listener)
2. Поле Email
3. Поле Password
4. Кнопка Login
5. Сообщение об ошибке (условно)

**Active Trips:**
1. Заголовок + счётчик активных рейсов
2. Панель фильтров (заказчик, статус)
3. Таблица рейсов
   - Строка рейса: trip_id | маршрут | заказчик | статус | время | кол-во алертов
4. Индикатор последнего обновления (polling)

**Alerts Feed:**
1. Заголовок + счётчик алертов
2. Панель фильтров (severity, status, заказчик, период)
3. Карточки алертов (sorted by: created_at DESC)
   - Карточка: severity badge | тип | trip_id | описание | время | кнопки действий
4. Пагинация / бесконечный скролл

**Trip Details:**
1. Breadcrumb: Active Trips / Alerts Feed → Trip {ID}
2. Карточка информации о рейсе (левая колонка)
3. История чатовых сообщений (правая колонка, скроллируемая)
4. Список алертов по этому рейсу (нижняя секция)

**Statistics:**
1. Выбор периода (сегодня / эта неделя / прошлая неделя / произвольный)
2. 4 KPI-карточки в ряд
3. Графики (2 колонки):
   - Рейсы по дням (line chart)
   - Задержки среднее время (bar chart)
   - Алерты по severity (stacked bar)

### 3.4 Пять UX-принципов для данного дашборда

#### Принцип 1: Критичное — всегда на виду

HIGH-алерты никогда не должны быть скрыты или погребены под другими данными. На любом экране, где присутствуют рейсы или алерты, элементы с severity=HIGH отображаются с красным визуальным маркером (#ef4444), который заметен без дополнительного внимания. Таблицы рейсов сортируются по умолчанию так, чтобы рейсы с HIGH-алертами стояли первыми.

**Обоснование:** Метрика M7 требует, чтобы Николя понимал алерт и знал что делать в 90% случаев. Если HIGH-алерт не виден сразу — метрика не достигается.

#### Принцип 2: Актуальность без действия пользователя

Данные обновляются автоматически (polling 3 сек). Пользователь не должен нажимать F5 или кнопку "Обновить". Индикатор последнего обновления показывает время последнего успешного запроса к API. При потере соединения отображается предупреждение, а не тихая деградация.

**Обоснование:** Метрика M2 требует latency ≤5 сек. Ручное обновление нарушает это требование для пользователя, который не смотрит на экран постоянно.

#### Принцип 3: Контекст всегда рядом с алертом

Каждый алерт содержит достаточно информации для первичного понимания ситуации: тип события, описание (текст от LLM), trip_id, заказчик, время. Переход к полному контексту (Trip Details) требует одного клика. Пользователь не должен делать более 2 кликов от алерта до истории чата.

**Обоснование:** Метрика M7 — Николя видит алерт и понимает что делать. Недостаточный контекст в карточке алерта ломает это требование.

#### Принцип 4: Статус-ориентированная модель данных

Каждый алерт имеет явный статус (new → reviewed → resolved), и этот статус отражён визуально. Пользователь понимает: "кто-то уже взял это в работу?" без дополнительных переговоров в Telegram. Действия смены статуса — одна кнопка, без многоуровневых диалогов подтверждения для reviewed, с простым диалогом для resolved.

**Обоснование:** В команде несколько менеджеров. Без явного статуса возможна ситуация когда несколько человек одновременно реагируют на один алерт, или наоборот — никто не реагирует, считая что другой уже занимается.

#### Принцип 5: Минимальный когнитивный груз

Дашборд используется в стрессовых ситуациях (HIGH-алерт в реальном времени). Интерфейс должен требовать минимума интерпретации:
- Цвет = severity (красный = HIGH, оранжевый = MEDIUM, зелёный = LOW)
- Иконки дублируют цвет (для людей с нарушениями цветовосприятия)
- Тексты алертов — конкретные и actionable, без технического жаргона LLM
- Фильтры с понятными подписями, сбрасываемые одной кнопкой

**Обоснование:** Cognitive overload в критической ситуации увеличивает время реакции. Каждая лишняя секунда на интерпретацию интерфейса = лишняя секунда задержки реакции.

### 3.5 Маппинг API → экраны

| Endpoint | Метод | Экран | Назначение | Частота |
|----------|-------|-------|-----------|---------|
| /api/auth/login | POST | Login | Аутентификация, получение JWT | При входе |
| /api/trips | GET | Active Trips | Список рейсов с фильтрами | Polling 3 сек |
| /api/trips/{trip_id} | GET | Trip Details | Детальная информация о рейсе | При открытии, polling 3 сек |
| /api/alerts | GET | Alerts Feed | Лента алертов с фильтрами | Polling 3 сек |
| /api/alerts | GET | Active Trips | Счётчик алертов по рейсу (badge) | Polling 3 сек |
| /api/alerts/{alert_id} | PATCH | Alerts Feed | Смена статуса reviewed/resolved | По действию пользователя |
| /api/alerts/{alert_id} | PATCH | Trip Details | Смена статуса из детали рейса | По действию пользователя |
| /api/stats | GET | Statistics | KPI-карточки | Polling 30 сек |
| /api/stats/chart | GET | Statistics | Данные для графиков Chart.js | При смене периода |
| /api/chats/{trip_id} | GET | Trip Details | История чатовых сообщений | При открытии + polling 10 сек |
| /api/health | GET | Любой экран | Индикатор соединения | Polling 10 сек |

**Примечание по polling:** Разные endpoint'ы имеют разные частоты. Active Trips и Alerts Feed — 3 сек (критичные данные). Statistics — 30 сек (агрегированные данные, не требуют мгновенного обновления). Health — 10 сек (статус соединения).

## 12. Примеры и граничные случаи

### 12.1 Граничные случаи таблицы рейсов

```
Случай 1: Рейс без алертов
- Строка: нормальный белый фон, нет badge-а, нет маркера ●
- Ячейка алертов: пустая или прочерк

Случай 2: Рейс с 0 active + 5 resolved алертами
- Алерты resolved не считаются "активными"
- Строка без красного фона
- Ячейка алертов: может показывать (5 решено) серым цветом

Случай 3: Рейс одновременно с HIGH и LOW
- Строка: HIGH-стиль (красный фон), badge HIGH первый
- Сортировка: данный рейс входит в HIGH-группу

Случай 4: trip_id не распознан системой (null)
- Отображается "ID не извлечён" в ячейке trip_id
- Строка кликабельна, Trip Details показывает что известно
```

### 12.2 Граничные случаи алертов

```
Случай 1: Алерт без описания от LLM (пустой текст)
- Карточка показывает тип алерта (delay, equipment_failure и т.д.)
- Вместо описания: "Подробности не извлечены системой"

Случай 2: Одновременный PATCH от двух менеджеров
- Второй PATCH получает ответ с актуальным статусом
- Polling обновит карточку через ≤3 сек
- Оптимистичное обновление откатывается если статус изменился

Случай 3: Более 100 алертов за день
- Пагинация или "Загрузить ещё" (инкрементальный скролл)
- Первые 20 алертов загружаются при открытии
```

### 12.3 Граничные случаи Statistics

```
Случай 1: Нет данных за выбранный период
- KPI-карточки показывают 0
- Графики показывают пустую область с надписью "Нет данных за период"

Случай 2: resolved_rate = 0% (все алерты open)
- KPI-карточка: красная, значение "0%"
- Не скрывать — это важный сигнал

Случай 3: HIGH-алертов = 0 за период
- KPI-карточка: зелёная (ниже нормы = хорошо)
- Не сообщать "ниже нормы" — это позитив

Случай 4: Chart.js получил пустой dataset
- График отрисовывается с пустыми осями
- Нет JavaScript ошибки
```
